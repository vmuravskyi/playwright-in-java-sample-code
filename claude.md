# Project Conventions

## Test Framework
- Playwright 1.57.0 for Java with JUnit 5 and Maven
- Run tests with `mvn verify` (failsafe plugin, not surefire)
- Java 17 with records and modern language features

## Playwright Lifecycle (two approaches)

### Preferred: `@UsePlaywright` annotation
Most tests use `@UsePlaywright(ChromeHeadlessOptions.class)`, which handles Playwright/Browser/BrowserContext/Page lifecycle automatically. `Page`, `BrowserContext`, etc. are injected as JUnit method parameters.

### Alternative: `PlaywrightTestCase` base class
Uses `ThreadLocal` to manage Playwright/Browser per-thread for parallel safety. Used by `LoginWithRegisteredUserTest`.

## Browser Configuration
- `ChromeHeadlessOptions` implements `OptionsFactory` — headless Chromium with `data-test` as the test ID attribute
- Locators should prefer `getByTestId()`, `getByRole()`, `getByLabel()`, and `getByPlaceholder()` over CSS selectors

## Page Objects
- Each page object takes a `Page` parameter in its constructor (no inheritance)
- Use `@Step` annotations from Allure on public methods
- Use `waitForResponse()` for network synchronization, not fixed sleeps

## Workflow Objects
- Higher-level abstraction above page objects for end-to-end tests
- Compose multiple page objects into business-level operations (e.g., `PurchaseWorkflow`, `AuthenticationWorkflow`)
- Also take `Page` in the constructor and instantiate their own page objects internally

## Interface Mixins for Cross-Cutting Concerns
- `WithTracing` — starts/stops Playwright traces per test, saves to `target/traces/`
- `TakesFinalScreenshot` — captures a full-page screenshot after every test
- `RecordsAllureScreenshots` — attaches screenshots to Allure reports
- Tests `implement` one or more of these interfaces to opt in

## Domain Models
- Use Java records: `User`, `Address`, `ProductSummary`, `CartLineItem`
- `User.randomUserNamed(firstName)` generates random test data via JavaFaker

## API-Backed Test Setup
- `UserAPIClient` uses Playwright's HTTP request API to register users via REST
- Prefer API calls over UI for test preconditions (faster, more reliable)

## Test Structure
- Use `@DisplayName` on test classes and methods
- Use `@Feature` and `@Story` Allure annotations for reporting
- Use `@Nested` classes to group related tests
- Use `@BeforeEach` to set up page objects and navigate
- Use AssertJ for assertions

## Package Organization (by feature area)
```
com.serenitydojo.playwright.toolshop
├── actions.api      — API clients (UserAPIClient)
├── catalog          — Product catalog tests + pageobjects/ + workflow/
├── contact          — Contact form tests + page objects
├── cucumber         — Cucumber runner, step definitions, fixtures
├── domain           — Domain records (User, Address)
├── fixtures         — Shared infrastructure (base class, mixins, options)
├── login            — Login tests + page objects
└── purchase         — End-to-end purchase tests
```

## Naming Conventions
- Test classes: `*Test.java`
- Page objects: descriptive nouns (`SearchComponent`, `ProductList`, `NavBar`)
- Workflow objects: `*Workflow.java`
- API clients: `*APIClient.java`
- Interface mixins: capability names (`WithTracing`, `TakesFinalScreenshot`)

## Cucumber BDD
- Runner: `CucumberTests` using `@Suite` + `@IncludeEngines("cucumber")`
- Step definitions reuse the same page object classes as JUnit tests
- `PlaywrightCucumberFixtures` manages Playwright lifecycle via `ThreadLocal` and Cucumber hooks
- `ScenarioTracingFixtures` records Playwright traces per Cucumber scenario
- Feature files in `src/test/resources/features/`

## Parallel Execution
- JUnit: class-level parallelism enabled, tests within a class run sequentially
- Cucumber: fixed parallelism of 4 scenarios
- Configured in `src/test/resources/junit-platform.properties`

## Reporting
- Allure report generated by `allure-maven` plugin after tests
- `@Step` annotations on page object and workflow methods create structured reports
- Traces saved to `target/traces/`, Allure results to `target/allure-results`
